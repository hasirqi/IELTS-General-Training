# 与用户共同确定的项目上下文

## 用户与目标

- 用户的英语词汇和句型基础薄弱，但懂中文、蒙古文和日文。解释首先使用清楚、自然的中文；日文和蒙古文只在确实能帮助记忆且关联准确时出现，不能把三种语言同时堆在页面上。
- 目标考试是 **IELTS General Training**，不是 Academic。目标周期为 6–9 个月，每天约 30–45 分钟，目标总分 6.0；产品内部以各单项不低于 5.5、连续两次模拟达到目标作为毕业参考。
- 产品不是背词表网站，而是一套能真正改善词汇、句型、听力、阅读、写作和口语的个人学习系统。产品口号是“英语难不倒我”，但口号应低调，实用学习入口永远优先。
- 用户重视直接执行和可见成果。不要反复确认已经明确的事情，不要用长篇计划代替实现。扩充内容时必须同步完成相关课程、句型和开口任务，不能只增加页面数字或只完成半套工作。

## 已确定的产品体验

- 这是面向成年人的实用主义产品。首页上来就是听力、阅读、写作、口语四个学习入口；不要恢复“今天学什么”“今天完成几步”之类的大型仪式化首页，也不要用虚假打卡、排行榜或幼稚激励制造压力。
- 全站顶部导航必须固定并出现在所有页面。品牌、词库入口和“向 ChatGPT 提问”保持一致；不要在课程区再添加重复、累赘的入口按钮。
- 中文导航、低信息密度、大触控区域、可调字号、清楚的焦点状态和明确的录音／播放状态是长期要求。桌面、Mac、iPhone、iPad 和安装式 PWA 都应可用。
- 发音质量很重要。正式听力课程必须使用仓库内固定的英式神经语音 MP3，不能把浏览器系统朗读当成考试听力。普通词条发音可以请求英语语音，但控件必须有无障碍名称。
- 听力同时提供学习模式和模考模式。学习模式可回放；模考模式只能播放一次、禁止拖动，提交后才显示原文。听力和阅读题干使用英语，答案解释用中文指出证据。
- 写作和口语的 ChatGPT 协作是手动闭环：复制结构化提示词、打开 `https://chatgpt.com/`、由用户把反馈粘贴回产品并本地保存。没有 API Key 时绝不能声称产品能读取 ChatGPT 网页、自动获得回答或自动评分。

## 学习系统的长期规则

- 不宣称存在“雅思全部单词表”。词库由高频基础词、General Training 生活与工作场景词、固定搭配和用户错误词构成，并持续检查覆盖缺口。
- 每个词分别训练形式、意义和使用。“看着眼熟”不算掌握。复习交替使用主动回忆、听音辨词、语境选义、填空、延迟回忆和口头使用。
- 中文释义采用“核心义优先＋高频常用义补充”：高频多义词应显示 2–4 个真正影响理解或表达的常用义，并标注容易误用的语境边界；不堆冷僻词典义。新词页的“认识”只代表词形和词义熟悉，不能直接计为使用掌握。
- “复习旧词”必须从真实学习状态动态生成：优先未解决错误和已到期内容，再根据遗忘调度、反应和掌握度安排；不能换成固定的十二个样本词。
- “新词与词块”使用已经通过内容门禁的词条，并由实际表现调整每日数量。错误内容必须重新进入复习。
- “句型实验室”需要展示主干、修饰、时态和连接关系，再让学习者换词、重组和造句；不能只提供一题长期不变的示例。
- “开口任务”必须覆盖实际生活沟通以及 Speaking Part 1、2、3，要求学习者真实开口、复述或回答；示范答案用于理解结构，不用于机械背诵。
- AI 或自动化不能自由决定课程范围，也不能修改客观题正确答案。课程框架、答案和评分依据必须是确定、可审计的。

## 内容进入完成数之前的门禁

- 新词必须严格对应 `lexicon.json` 的下一批源词顺序。每条都要有聚焦中文义、包含目标词的自然英文句、包含目标词的常用搭配、场景分类和批次标记。
- 单词例句至少五个英文词并有完整标点；含义应聚焦，不能用整段百科解释。例句和搭配在所有已审核批次中必须唯一。
- 新课程必须有唯一 ID、全新标题、真实且不同的场景。听力和阅读每节均需四道三选一客观题，答案必须存在于选项中且可从原文直接验证。
- 写作必须覆盖 General Training Task 1 的正式、半正式和私人书信以及 Task 2；口语必须持续覆盖 Part 1、2、3。不要把 Academic Writing Task 1 图表题混入产品。
- 新听力课必须同时交付与转写一致的固定英式 MP3，并检查文件存在、格式正确且不是异常小文件。
- 句型题必须有三个不同选项、唯一有效答案、中文解释、错误提示和完整示范句。开口任务的题目、场景和示范不得与旧内容重复。
- 每一批在计入完成数前都要检查：源词顺序、跨批重复、答案有效性、语境自然度、占位词、课程结构、音频映射和构建结果。发现重复应改写内容，不能降低测试标准。
- 当用户要求“同步补充所有新的课程场景”时，词库审核、听说读写课程、句型实验室和开口任务要作为一个完整批次交付。

## 技术与数据边界

- 技术栈为 React、TypeScript、Vite 和可安装 PWA；学习状态使用 IndexedDB 本地保存。核心学习不依赖 OpenAI API Key。
- 本地保存意味着换电脑或换浏览器不会自动带走旧学习进度。除非未来明确实现同步，否则不要暗示已有跨设备学习记录同步。
- GitHub 仓库：`https://github.com/hasirqi/IELTS-General-Training`
- 线上地址：`https://hasirqi.github.io/IELTS-General-Training/`
- `main` 分支推送后由 GitHub Actions 自动测试、构建和发布 GitHub Pages。
- 常用验证命令为 `pnpm test` 和 `pnpm build`。实现后由代理自己启动本地服务器并打开预览，不要在可以代为执行时只把启动命令丢给用户。

## 工作方式与 Git 纪律

- 用中文沟通，先说结果和证据，再补必要解释。语气直接、耐心、不羞辱，也不要把用户当小孩。
- 实现任务要持续做到可验证的完成状态：编辑、测试、构建、必要时本地预览。若用户要求推送，则还要提交、推送、等待 GitHub Actions 完成并核对线上资源。
- 报告内容工作时给出精确数字，例如审核词条数、各技能课程数、题目数、音频数和测试通过数；不要使用“应该有”“大概完成”一类说法。
- 工作区可能包含用户自己的未跟踪文件。当前 `audit/` 是本地审核材料，不得擅自加入提交。使用明确的 `git add` 文件列表，避免 `git add .`。
- 不覆盖用户已有修改，不使用破坏性 Git 命令。提交前运行内容测试、生产构建、`git diff --check`，并确认暂存区只包含本轮内容。
